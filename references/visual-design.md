# Visual Design Guide（投影片視覺憲法）

> 這份是「**投影片怎麼做才圖文並茂、不變字牆**」的單一入口，跟 `teaching-style.md`
> （怎麼教）、`narration-style.md`（怎麼講）互補。三份一起看。
> 工程細節（字級下限、燒字幕、組裝）看 `SKILL.md`。**這份只管視覺設計。**
>
> 為什麼需要這份：這套 skill 本來只規範「字要多大、畫面要填滿」，卻沒有任何一條規定
> 「投影片上要有圖」。結果一張「大標題 + 一段置中大字」可以通過所有檢查 —— 那正是字牆。
> 這份把「每張都要圖文並茂」變成可檢查的硬規則。

---

## 0. 第一原則：投影片是「畫重點」，不是「貼字幕」

**母規則**：**旁白負責把故事講完，投影片負責給觀眾『一張看得懂的圖』。**

觀眾同時在「聽」和「看」。如果投影片只是把旁白打成字，觀眾的眼睛就沒事做 ——
要嘛放空、要嘛低頭滑手機。投影片的價值，是給耳朵聽到的東西**配一個畫面**：
一個流程、一個對比、一條長條、一張真截圖、一個放大的數字。

> 開做投影片前的自我測試：把每一張的**文字全部遮掉**，只看圖，還能猜出「這張在講什麼」嗎？
> 猜不出來 = 這張沒有視覺，只是字牆。

---

## 1. 每張投影片都要有一個「視覺錨點」（硬規則）

**每一張投影片都必須有一個非文字的主視覺**，七選一：

| 視覺錨點 | 什麼時候用 |
|---|---|
| **流程圖** | 有先後順序、A 變成 B、步驟 |
| **對比圖** | 兩件事對照：前/後、對/錯、A/B、失敗群/成功群 |
| **長條圖 / 比例圖** | 幾個數字比大小 |
| **放大數字** | 單一個關鍵數字或比例 |
| **真實截圖** | 「我看了／做了／跑了 X」（teaching-style §3 強制） |
| **比喻場景 / 圖解** | 抽象概念，用一個具體畫面或簡圖說明 |
| **時間軸** | 發展過程、案例的起源→經過→結局 |

**判準（可自我檢查）**：
- 說得出這張用的是上面哪一種 → 過。
- 說不出來、只有「標題 + 幾行字」 → **不合格，重做那一張**。
- 一整支影片每張都是同一種錨點（全部長條圖）也要警覺，換點花樣。

> Path C（`generate_slides.js` 純文字條列）天生做不到這條，所以它只是**沒有其他辦法時的
> 緊急備案**，不能拿來做正式頻道影片。

---

## 2. 文字預算：投影片上的字要少（硬規則）

一張投影片的可見文字上限：

- **一個標題**（≤ 15 字）
- **＋ 最多 3 個短點**，或幾個短標籤（每點 ≤ 15 字）
- **正文可見字數 ≤ 約 60 個中文字**

超過就是塞太多。處理方式：

1. **拆成兩張**投影片（一張一個概念），不是把字級縮小。
2. 把細節**移回旁白**（旁白本來就該講完整句子）。
3. 長句 → 濃縮成**關鍵詞**放在圖旁邊當標籤。

> 「一個概念 = 一張投影片 = 一個視覺」。塞第二個概念進來，就該開新的一張。

---

## 3. 圖 > 字：能畫就不要寫

呼應 `teaching-style.md` §2「流程圖比文字定義好」、§8a「用長條圖視覺顯示差距」。
把常見的「用文字寫」改成「用圖畫」：

| 你想寫的文字 | 改成畫的圖 |
|---|---|
| 「第一步…第二步…第三步…」 | 流程圖（`layout-flow`） |
| 「A 有這些缺點，B 有這些優點」 | 左右對比（`layout-compare`） |
| 「我的分數 48，別人 72、95」 | 長條圖（`layout-bars`） |
| 「準確率高達百分之九十五」 | 放大數字（`layout-bignumber`） |
| 「一開始…後來…現在…」 | 時間軸（`layout-timeline`） |
| 一段名詞的定義 | 一句白話 gloss + 一個簡圖／圖示 |

## 4. 旁白裡的數字與比較，一律圖像化（硬規則）

**旁白只要出現「數字」或「比較」，投影片上就要用圖呈現，不能只用一句話帶過。**

- 單一數字 → 放大數字版型（`layout-bignumber`），配一句解讀當錨點
  （呼應 teaching-style §1a：先給直覺，再給數字）。
- 多個數字比大小 → 長條圖（`layout-bars`）。

### 誠實長條圖規則（呼應 teaching-style §8a）

小樣本、拿自己跟別人比的時候，**用圖誠實顯示差距，不要用文字假裝差不多**：

- 長條長度 = 真實數值比例，**不縮放、不誇大**。
- 座標**從 0 開始**，不截斷（截斷會讓差距看起來比實際大或小）。
- 每條長條**直接標數值**，不要另外畫看不懂的圖例。
- reframe 成 0–100 分之後，**再 factcheck 一次**數字有沒有對（teaching-style §1d）。

---

## 5. 「我做了 X」型影片：一定要真截圖（呼應 teaching-style §3）

「我看了／做了／跑了 X」型影片，投影片**必須嵌入該 X 的真實截圖**，
不能用文字、emoji、或自己畫的示意圖代替 —— 觀眾看到真畫面才會信「他真的做了」。

- 把畫面存到專案的 `screenshots/` 資料夾。
- 用 `layout-screenshot.html`：`<img>` 相對路徑指過去，需要時疊一個紅框圈重點。
- **這是 Path B（HTML）的專屬能力**：Path A（gpt-image-2 生圖）做不出真畫面。
  所以 §3 型影片優先選 Path B；若主體用 Path A，這幾張截圖投影片改用 HTML/合成，
  但要 pad 成同樣的 1410×940 + 140px 字幕帶幾何，整支風格才一致。

---

## 6. 視覺 ≠ 違反「投影片與旁白對齊」

`SKILL.md` 有一條「投影片上每個元素都要在旁白提到，反之亦然」（來自 lessons #7）。
這條是要抓**沒人念到的孤兒條列點**，**不是**要禁止圖。

- 流程圖的方框標籤、長條圖的座標、比喻場景裡的元素 —— 只要它們是在**圖解旁白正在講的
  那個概念**，就算「有提到」，不必逐字念出每個標籤。
- 真正要對齊的是：**投影片上的每個「主張／數字」都要有旁白覆蓋**，反過來旁白講的重點
  也要在投影片看得到。圖是用來「表達」那個重點的，不是多出來要另外交代的東西。

## 7. Path A（gpt-image-2 手繪風）視覺 prompt 樣板庫

Path A 靠文字 prompt 叫模型畫圖。**每一個 prompt 都要明確指定「畫什麼構圖」，
不能只給「標題 + 幾行字」**（那樣模型就真的只畫一張字卡）。

用法：把下面的骨架填空，接在共用 style block 後面（`slides_prompts.json` 用
`{"style": ..., "slides": [...]}` 格式，style 只寫一次）。所有要出現的字都用「」框住，
並沿用既有防雷規則（不要真人臉、不要亂編數字、白底手繪、中文正確）。

**共用結尾（每個 prompt 都加）**：
`「所有中文字必須完全正確、清楚可讀、不可有亂碼或錯字。數字要正確，畫面只能出現我指定的數字，其他不要亂加。」`

### 7a. 流程圖
```
標題（左上加底線）：「＿＿」。中央一條水平流程：
方框「＿＿」→（黑色細箭頭）→方框「＿＿」→（箭頭）→方框「＿＿」。
每個方框下方一行小字說明。大量留白。
```

### 7b. 左右對比
```
標題（左上加底線）：「＿＿」。畫面用一條垂直線分成左右兩半：
左半上方標「＿＿」，下面畫＿＿（簡圖）；右半上方標「＿＿」，下面畫＿＿。
中間一個圈寫「VS」（或畫箭頭表示左變成右）。
```

### 7c. 長條圖（誠實、指定數值）
```
標題（左上加底線）：「＿＿」。畫面中央一組手繪長條圖，白底、黑色座標軸從 0 開始。
共＿條長條，由左到右標籤與高度：「＿＿」約＿、「＿＿」約＿、「＿＿」約＿。
每條長條頂端標上它的數值。畫面只能出現這幾個數字，不要多加任何數字或多畫長條。
```

### 7d. 時間軸
```
標題（左上加底線）：「＿＿」。中央一條水平時間軸（左到右一條線加幾個圓點）。
節點由左到右：「＿＿＿」、「＿＿＿」、「＿＿＿」，每個節點上方標時間、下方一行小字。
```

### 7e. 放大鏡 / 剖面標註圖
```
標題（左上加底線）：「＿＿」。中央畫一個＿＿，旁邊用一個放大鏡圈住其中一部分放大，
用黑色細線把放大處拉出來標註：「＿＿」、「＿＿」。手繪、白底、大量留白。
```

### 7f. 比喻場景
```
這是比喻頁。用一個生活場景畫出抽象概念：畫＿＿（具體的日常畫面），
旁邊用簡單線條和一兩個「」標籤點出對應關係。圓頭火柴人，不要真人臉。
```

### 7g. 資料表格（最容易被模型亂填，嚴格限制）
```
標題（左上加底線）：「＿＿」。畫一個＿列＿欄的手繪表格。
只填以下幾格，其餘格子留空或畫橫線：（列出每一格的確切內容與位置）。
背景留白、不要重複任何文字、不要在空格裡自己編數字或字。
```

> 生成後**每張都要用眼睛檢查**：中文有沒有亂碼、有沒有自己多編數字、構圖對不對；
> 錯的那張單獨重生（gpt-image-2 的雷詳見 `lessons-learned.md`）。

## 8. 每張投影片的自我檢查（做完一張問一次）

```
□ 遮掉文字只看圖，猜得出這張在講什麼嗎？（猜不出＝沒視覺）
□ 說得出這張的視覺錨點是七種的哪一種嗎？
□ 可見文字 ≤ 一個標題 + 3 短點（≤ 約 60 中文字）嗎？
□ 旁白裡的數字／比較，有沒有變成圖（長條／放大數字／對比）？
□ 「我做了 X」型的，有沒有放真截圖（而不是示意圖）？
□ 正文字級 ≥32px、主標 ≥72px，手機上看得清楚嗎？
□ 這張跟前一張的版型不一樣嗎？（連續同版型會膩）
```

## 開拍前 & 收尾 checklist（濃縮成一頁）

```
【規劃階段（寫完 narration 後、生圖前）】
□ 幫每一張投影片寫下「這張要畫什麼」＋「用哪種視覺錨點」（storyboard）
□ 說不出某張的圖 → 那段旁白是資訊傾倒，回去重寫
□ Path B：對照 slide-templates/README.md 挑版型
  Path A：對照 §7 挑 prompt 樣板，每個 prompt 都指定構圖

【生成後（視覺驗證）】
□ 每張都有視覺錨點，沒有純字牆（有 → 重做那張）
□ 每張文字都在預算內
□ 旁白的數字／比較都圖像化了
□ 誠實長條圖：比例真實、座標從 0、直接標值、reframe 後再 factcheck
□ 「我做了 X」型：真截圖已嵌入
□ 抽一張 verify.png 再問一次：這是圖，還是字牆？
```
